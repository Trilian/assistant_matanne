🏆 Notes détaillées par catégorie

8.5

🏗️ Architecture

85%

Forces: Architecture modulaire exemplaire — séparation claire routes/schemas/services/models. Factory pattern (@service_factory) pour DI propre. Event bus pub/sub pour découplage. Package structure cohérente dans core/ (10 sous-packages). Décorateurs réutilisables (@avec_session_db, @avec_cache, @avec_resilience). 15 packages services bien isolés.

Faiblesses: admin_operations.py (~850 lignes, 28 endpoints) est un fourre-tout. inter_modules.py (~1200 lignes) gagnerait à être splitté par domaine. Quelques aliases dépréciés dans jeux/__init__.py polluent les imports.

Pour atteindre 10/10: Splitter les 2 fichiers monolithiques. Nettoyer les aliases legacy. Documenter les patterns architecturaux dans un ADR (Architecture Decision Records).

8.5

⚙️ Backend API

85%

Forces: 500+ endpoints REST cohérents. Patterns standardisés (@gerer_exception_api + Depends(require_auth) + executer_avec_session). JWT auth + rate limiting (60/min standard, 10/min IA). ETag cache middleware. Pagination cursor-based. WebSocket courses temps réel. 6 middlewares empilés. Documentation Swagger automatique.

Faiblesses: Pas de retry/backoff pour les notifications échouées. Event bus non persisté en DB (perdu au restart). Pas de tracking granulaire des tokens IA par service.

Pour atteindre 10/10: Implémenter retry + dead letter queue pour les notifications. Persister les events en DB. Ajouter compteur tokens IA par service. Webhook Telegram entrant pour commandes utilisateur.

8.5

🤖 IA & Automations

85%

Forces: 15+ services IA (Mistral) avec BaseAIService abstrait. Cache sémantique + circuit breaker + rate limiting intégrés. 25+ bridges inter-modules via event bus. 20+ cron jobs planifiés avec couverture complète. Suggestions adaptatives (saison, feedback, stock). 5 canaux de notification (Telegram principal).

Faiblesses: Pas de tracking tokens par feature (coût opaque). Event bus mémoire uniquement. Pas de webhook Telegram entrant (envoi seul). Quelques bridges non encore wirés au frontend.

Pour atteindre 10/10: Token tracking par service. Persistance events. Commandes Telegram entrantes (/planning, /courses). IA contextuelle enrichie dans chaque module.

8

🔐 Sécurité

80%

Forces: JWT Bearer + require_auth sur tous les endpoints. RLS policies PostgreSQL (30+). Rate limiting middleware. Input sanitization (XSS/injection via sanitizer.py). Security headers middleware. CORS restrictif. Audit OWASP documenté. Security logs.

Faiblesses: Pas de CSP (Content Security Policy) strict. Canal email non sécurisé (pas encore configuré). Pas de rotation automatique des JWT secrets.

Pour atteindre 10/10: CSP strict headers. Rotation JWT secrets. Audit de sécurité automatisé dans CI. Rate limiting par IP + user combiné.

7.5

🧪 Tests & Qualité

75%

Forces: 257 fichiers de tests, 1000+ tests. Pytest + Vitest + Playwright. Contract tests, mutation testing, benchmarks, load tests. SQLite in-memory fixtures pour isolation. Tests E2E configurés.

Faiblesses: 31 tests skippés (13 batch_cooking DB mock, 10 AI, 8 divers). Webhook Telegram non testé. Notification retry logic non testée. Quelques modules sous-testés.

Pour atteindre 10/10: Fixer les 31 tests skippés. Ajouter tests webhook Telegram. Ajouter tests notification dispatcher. Coverage > 90% sur tous les modules critiques.

7.5

⚡ Performance

75%

Forces: Cache multi-niveaux (L1 mémoire + L3 fichier) via @avec_cache. ETag HTTP middleware. TanStack Query staleTime côté frontend. Lazy loading widgets dashboard. QueuePool SQLAlchemy. Prometheus metrics.

Faiblesses: Pas de Redis (L2 cache). Pas de CDN pour les assets statiques. Railway Free tier limité en mémoire. Pas de connection pooler externe (PgBouncer).

Pour atteindre 10/10: Redis L2 (pas possible Free). CDN Vercel pour assets (déjà partiellement). Optimiser les requêtes N+1. Profiling régulier.

7.5

🎨 Frontend UI

75%

Forces: 48+ composants UI shadcn/ui. Storybook v10 configuré. 18 widgets dashboard drag-drop. Plan 3D Three.js. 13 composants graphiques (Recharts/D3). Responsive mobile/tablette. Dark mode. PWA. Heatmaps, Sankey, Radar, Timeline, Kanban.

Faiblesses: Manque skeleton loaders sur toutes les pages. Empty states non illustrés. Transitions de page absentes. Micro-animations limitées. Pas d'onboarding premier lancement.

Pour atteindre 10/10: Skeleton loaders partout. Empty states SVG. Transitions fluides. Micro-animations jauges. Onboarding interactif. Search global (cmdk).

7.5

🚀 DevOps / CI/CD

75%

Forces: 8 workflows GitHub Actions (backend-tests, frontend-tests, mutation-tests, deploy, deploy-preview, deploy-production, db-backup). Docker + docker-compose staging. manage.py 30+ commandes. Dependabot configuré. Pre-commit hooks.

Faiblesses: Pas de rollback automatique en production. Pas d'environnement de staging automatisé (preview oui, staging permanent non). Pas de monitoring des workflows eux-mêmes.

Pour atteindre 10/10: Rollback automatique si health check échoue. Staging permanent. Alertes si workflow échoue (Telegram).

7

📚 Documentation

70%

Forces: 42 fichiers docs. Audits OWASP, Lighthouse, A11Y, UX 3-clic. Architecture, API Reference, Patterns, Services Reference, Data Model. CHANGELOG, DEPRECATED, ADMIN, AUTOMATION guides.

Faiblesses: docs/user-guide/ non mis à jour pour l'UI actuelle. SERVICES_REFERENCE.md avec quelques chemins cassés. Pas de doc unifiée pour le système de bridges. Pas de doc API pour les endpoints admin.

Pour atteindre 10/10: Mettre à jour user-guide. Documenter API admin dans ADMIN.md. Doc unifiée bridges. Corriger SERVICES_REFERENCE.

7

🎯 UX / Ergonomie

70%

Forces: Audit 3-clic OK. Responsive mobile + tablette. PWA installable. Dark mode auto. Navigation par hub → sous-pages. Bottom bar mobile. Breadcrumbs. 18 widgets dashboard personnalisables.

Faiblesses: Pas d'onboarding/tutoriel premier lancement. Pas de search global. Quelques flux en 4+ clics. Navigation entre modules sans transitions. Pas de mode "quick capture" (ajout rapide).

Pour atteindre 10/10: Onboarding interactif. Search global cmdk. Quick capture flottant. Bottom sheet mobile. Transitions fluides.

📐 Principes directeurs

👨‍👩‍👦 L'utilisateur est pilote

L'IA propose, l'humain valide. Jamais d'action automatique sur les données importantes sans confirmation.

🔒 Admin invisible

Les fonctions admin (jobs manuels, dry-run, console) restent cachées de l'UI utilisateur standard.

🎮 Budget jeux séparé

Ne jamais fusionner avec le budget famille. C'est une décision volontaire.

🚫 Pas de social/communautaire

Pas de partage public, pas de marketplace. App familiale privée.

🍽️ Bien manger avant tout

Jamais de mode "économies alimentaires". Se faire plaisir est prioritaire.

🚂 Railway Free

Optimiser la mémoire backend. Pas de plan payant. Contrainte à respecter.

📱 Telegram, pas WhatsApp

100% gratuit, messages illimités, boutons interactifs. Canal conversationnel principal.

👶 Jules = repas famille adaptés

Pas de menu séparé — version adaptée (sans sel, mixé, pas d'alcool). Pas de courbe de croissance OMS.

Principes UX

  • Flux en 3 clics max — Toute action courante accessible en ≤3 taps
  • Pas de surcharge — Dashboard personnalisable (masquer/ajouter widgets)
  • Feedback immédiat — Toast confirmation sur chaque action
  • Validation explicite — Pas d'action auto sans confirmation pour données importantes
  • Mobile first — Tout doit être utilisable sur tablette Google

Contraintes techniques rappelées

  • Railway Free tier — optimisation mémoire obligatoire, pas de Redis
  • Pas de scan tickets / OCR — trop contraignant, décision définitive
  • Pas de courbe croissance Jules / références OMS poids/taille/IMC
  • Pas de gamification familiale (sauf sport/Garmin uniquement)
  • Pas d'alertes vaccins auto avant 6 ans
  • Pas de push géolocalisation (habite à côté du supermarché)
  • Pas de RGPD/conformité (app familiale privée)
  • Pas de saisie prix manuelle à chaque achat
  • Google Assistant uniquement (pas Siri — pas Apple)
