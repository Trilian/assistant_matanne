🏗️ 1. Architecture & structure des fichiers

Vue d'ensemble

Le module Jeux est organisé en 3 sous-modules (Paris Sportifs, Loto, EuroMillions) avec un dashboard agrégé, un gestionnaire de bankroll et une page performance. L'architecture suit un pattern de router agrégé.

🐛 3. Bugs & problèmes identifiés

#SévéritéProblèmeFichierImpact
1 HAUTE sync_jeux_budget contredit l'architecture
Le cron job _job_sync_jeux_budget() écrit les gains/pertes jeux directement dans la table BudgetFamille. Cela contredit la décision architecturale documentée partout (DEPRECATED.md, user-guide, préférences utilisateur) que les budgets jeux et famille sont séparés. 		src/services/core/cron/jobs.py L2166-2227
src/services/core/cron/jobs_schedule.py L119
src/api/routes/jeux_paris.py L818 		Les gains/pertes paris sont insérés comme lignes dans BudgetFamille chaque nuit à 22h. Pollution du budget familial.
2 MOYENNE BudgetResponsableResponse — vestige "jeu responsable"
Le schema Pydantic contient encore cooldown_actif, auto_exclusion_jusqu_a, limite — tous des attributs du module "jeu responsable" pourtant supprimé du codebase. Le DashboardJeuxResponse inclut un champ budget: BudgetResponsableResponse. 		src/api/schemas/jeux.py L419-433 Données fantômes renvoyées dans le dashboard (toujours valeurs par défaut). Confusion pour les consommateurs de l'API.
3 BASSE 12 self-assignments circulaires
Pattern obtenir_x = obtenir_x présent dans tous les services _internal/. C'est un no-op complet — la variable est réassignée à elle-même. 		src/services/jeux/_internal/*.py (12 fichiers) Aucun impact fonctionnel. Code mort qui ajoute du bruit.
4 BASSE Encodage UTF-8 corrompu dans commentaires
Les séparateurs de sections dans les routes utilisent des caractères box-drawing mal encodés (â•â•â• au lieu de ═══). 		src/api/routes/jeux_paris.py, jeux_dashboard.py, jeux_euromillions.py Visuel uniquement. Le code fonctionne correctement.
5 MOYENNE Validation Zod manquante pour Loto et EuroMillions
Seul schemaPariSportif existe côté frontend. Il manque les schémas Zod pour la création de grilles Loto (5 numéros 1-49 + chance 1-10) et EuroMillions (5 numéros 1-50 + 2 étoiles 1-12). Les formulaires front envoient des données sans validation client. 		frontend/src/bibliotheque/validateurs-jeux.ts La validation Pydantic backend rattrape les erreurs, mais l'UX est dégradée (erreurs serveur au lieu de feedback instantané).

Détail Bug #1 — sync_jeux_budget

# src/services/core/cron/jobs.py L2166-2227
def _job_sync_jeux_budget() -> None:
    """JOB-2 — Synchronise les gains/pertes jeux vers le budget famille."""
    ...
    with obtenir_contexte_db() as session:
        mises = session.query(func.sum(PariSportif.mise))...
        gains = session.query(func.sum(PariSportif.gain))...
        net = float(gains) - float(mises)

        # ⚠️ ÉCRIT DIRECTEMENT DANS BudgetFamille !
        session.add(BudgetFamille(
            date=today - timedelta(days=1),
            categorie="jeux",
            description="Sync jeux auto",
            montant=abs(net),
        ))
        session.commit()
Recommandation Bug #1 : Supprimer le cron job sync_jeux_budget et sa planification. Le budget jeux a sa propre table jeux_bankroll_historique et le dashboard unifié (/api/v1/dashboard/budget-unifie) agrège déjà les données des deux budgets en lecture seule.
Fichiers à modifier : jobs.py, jobs_schedule.py, admin_constants.py, jeux_paris.py (marqueur sync).

🔄 5. Redondances & duplication

Duplication Loto ↔ EuroMillions (~70% similaire)

Les modules Loto et EuroMillions partagent une grande partie de leur logique :

FonctionnalitéLotoEuroMillionsSimilitude
CRUD tirages/grillesloto_crud_service.pyeuromillions_crud_service.py90%
Statistiques fréquencesloto/statseuromillions/stats85%
Heatmap composantHeatmapNumerosHeatmapNumeros (même)100%
Backtestbacktest_service.pyMême service paramétré80%
Génération IAgenerer-grille-ia-pondereegenerer-grille-ia70%
Page frontendloto/page.tsx (~500 lignes)euromillions/page.tsx (~600 lignes)65%
Modèle ORMTirageLoto + GrilleLotoTirageEuromillions + GrilleEuromillions75%
Constat : La duplication Loto/EuroMillions représente ~2 000 lignes de code redondant. Un refactoring vers une base commune (abstract LotteryService, composant LoteriePage paramétrable) pourrait réduire de ~40% le code.
Mais : Ce n'est pas urgent. Les deux modules fonctionnent indépendamment et leur séparation facilite l'évolution indépendante (ex: étoiles EuroMillions vs numéro chance Loto).

Autres redondances mineures

  • Dashboard KPIs : calculés dans jeux_dashboard.py ET dérivables depuis stats_personnelles.py — double source de calcul ROI
  • Forme des équipes : calculée dynamiquement dans la route jeux_paris.py ET stockée dans la colonne forme_recente du modèle Equipe
  • Stats Loto : cache StatistiquesLoto (table SQL) ET calcul à la volée dans les routes — pas toujours synchronisés
Verdict Cohérence / Redondances : 6/10 — Duplication significative Loto/EuroMillions, vestiges jeu responsable, double calcul de certaines métriques. Fonctionnel mais peu DRY.
